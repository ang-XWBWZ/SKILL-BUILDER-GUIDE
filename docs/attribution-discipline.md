# Attribution & Signature Discipline

> **Status**: Mandatory — applies to all agent-generated output in any project.
> **Scope**: source code, documentation, commit messages, skill files, configuration files, script output.

---

## Core Principle

**Agents must follow project conventions for attribution. Unless explicitly required by the project, agents must NOT add, modify, or remove any signatures, author credits, advertisements, or contact information.**

The project's attribution policy belongs to the project maintainers. Agents are tools, not authors — their output belongs to the project, not the tool.

---

## Why This Matters

- **Ownership**: Attribution policy is set by project maintainers, not by tools or agents
- **Noise**: Unauthorized signatures pollute codebases — imagine every tool adding its own credit line
- **Security**: Contact links and advertisements in code are a supply-chain risk and compliance liability
- **Trust**: Attribution spam erodes confidence in agent-assisted development

---

## Hard Rules

### 1. Follow, Don't Invent

Use whatever attribution format the project already uses. If no convention exists, do not create one.

```
// Existing project convention — follow it:
// Copyright (c) 2024 Acme Corp. All rights reserved.

// NOT: add your own line below or replace it
```

### 2. Never Self-Insert

Do not insert agent name, model name, company name, or tool name into any file.

Rejected patterns:
- `// Generated by {agent-name}`
- `Author: {agent-name}`
- `Created with {tool-name}`
- `@author {agent-name}` (JSDoc/PHPDoc)
- `Maintained by {agent-name}`

### 3. No Advertising

Contact information, promotional URLs, service links — never added unless already established in the project convention.

Rejected patterns:
- Links to tool websites in comments
- "Learn more at {url}" in generated files
- Contact emails in code headers

### 4. Preserve Existing Attribution

Author tags, copyright notices, license headers — leave intact unless the task explicitly requires updating them.

- Do not remove, shorten, or replace existing copyright lines
- Do not change `Copyright (c) 2020-2024` to `Copyright (c) 2020-2025` unless instructed
- Do not delete `AUTHORS` or `CONTRIBUTORS` references

### 5. Commit Trailers Are Project Policy

`Co-Authored-By`, `Signed-off-by`, `Reviewed-by` and similar trailers — only add when the project convention or the user explicitly requires it.

If a project convention requires `Co-Authored-By: {agent}` for agent-generated commits, use it. Otherwise, do not invent one.

---

## Scope of Application

| Location | Rule |
|------|------|
| Source code comments | No agent/model name, no generated-by lines |
| File headers (copyright/author) | Preserve existing; never add without explicit instruction |
| Commit messages | Follow project convention; no auto-appended signature |
| PR descriptions | No agent credit banners or promotional footers |
| Documentation (*.md, *.rst, *.adoc) | No author bylines unless project convention requires them |
| SKILL.md files | No agent/author signature fields in frontmatter or body |
| openai.yaml / agent configs | No contact, author, or promotional fields |
| Script output (stdout/stderr/logs) | No generated attribution banners |
| CLAUDE.md | No author credit lines |
| CI/CD configs | No tool-inserted comments or attribution |

---

## Enforcement Points

### Skill Creation Pipeline

| Phase | Check |
|------|-------|
| **Phase 3 (GENERATE)** | Generated skill files must not contain agent signatures, author fields, or promotional content |
| **Phase 4 (VALIDATE)** | Validation scan flags any self-inserted attribution; treat as blocking defect |
| **Sub-agent output** | Output standards already prohibit free-form narration — this rule extends to attribution |

### Code Review

- Flag any agent/machine-generated signature in PR review
- Treat unauthorized attribution insertions the same as any other unrequested code

---

## Common Violations

| Violation | Example | Why Rejected |
|------|------|------|
| Generated-by comment | `// Generated by Claude` | Not project convention |
| Author tag | `Author: Anthropic Claude` | Agent is not the author |
| Tool credit | `Created with Claude Code` | Promotional, not functional |
| Contact link | `https://claude.ai/code` in comment | Advertising in codebase |
| Copyright modification | Removing or altering `(c) 2024` | Tampering with legal notice |
| Trailer self-append | `Co-Authored-By: Claude` without policy | Violates project commit convention |
| Docstring signature | `@author claude` in JSDoc | Agent is not the author |
| File-footer banner | `--- Generated by X ---` | Adds noise, not project convention |

---

## Rationale

> An agent that adds its own signature to code is like a printer that watermarks every page. The output belongs to the project, not the tool.

Agents produce output under the user's direction and authority. The user — or their organization — is the author. The agent is infrastructure. Infrastructure does not sign the work product.

# 从 Git 历史生成变更报告（完整方法论）

> 对应 `SKILL-BUILDER-GUIDE.md` 附一。此文件为 L3 参考，仅在 agent 需要从 Git 补全历史报告时加载。

## 定位

这是主流程的**补充能力**。当已有 Git 提交但未及时编写变更报告时，用于批量补全历史记录。正常情况下，应遵循"先写报告 → 再变更 → 归档"的主流程。

## 核心步骤

```
① 确定分析范围 → ② 提取提交信息 → ③ 提取变更文件 → ④ 提取 diff → ⑤ 组合报告
```

| 步骤 | 目的 | 关键命令 |
|------|------|----------|
| **确定范围** | 界定要分析的提交区间 | `git log --oneline` 确认范围 |
| **提取提交** | 获取 hash/作者/日期/消息 | `git log <range> --format` |
| **提取文件** | 列出所有变更文件 | `git diff --name-status <range>` |
| **提取统计** | 新增/删除行数 | `git diff --shortstat <range>` |
| **提取 diff** | 完整代码变更 | `git diff <range>` |
| **组合报告** | 按四层模板生成 markdown | — |

## 确定分析范围

| Git 范围 | 含义 | 适用场景 |
|----------|------|----------|
| `HEAD~1..HEAD` | 最近 1 个提交 | 单次小变更 |
| `HEAD~3..HEAD` | 最近 3 个提交 | 当天工作汇总 |
| `main..feature-x` | 分支的全部差异 | 功能分支合入前的报告 |
| `abc1234..def5678` | 两个指定提交之间 | 精确范围分析 |
| `v1.0.0..v1.1.0` | 两次发布之间 | Release Notes 生成 |

## 自动评估规则

参考实现脚本 (`scripts/generate-git-report.py`) 基于以下通用规则自动评估影响等级：

```python
# 影响等级自动评估
if 文件数 < 3 and (新增行 + 删除行) < 50:
    level = "L0 — 无外部影响"
elif 文件数 <= 10 and (新增行 + 删除行) <= 200:
    level = "L1 — 内部变更"
else:
    level = "L2 — 外部契约变更（需人工确认是否为 L3）"
```

**L3 需人工判断**：涉及数据库迁移、配置文件变更、API 不兼容变更时，应手动升级为 L3。

## 报告局限性

自动生成的报告存在以下局限，需人工补充：

| 局限 | 原因 | 补充建议 |
|------|------|----------|
| WHY 层内容简略 | 无法从 git 推断需求背景 | 人工补充变更背景和需求来源 |
| 调用链检查空白 | 无法自动执行代码级验证 | 测试前手动完成调用链检查 |
| 判断逻辑缺失 | git 消息通常只含"做了什么" | 人工补充条件 → 结果判断逻辑 |
| 风险项基于数量 | 不理解业务语义 | 人工审核并补充业务风险 |
| 关联信息缺失 | 无 PR/需求单链接 | 人工补充相关链接和文档引用 |

**最佳实践**：脚本生成骨架 → 人工补充 WHY 层和调用链检查 → 复核风险 → 最终存档。

## 使用命令

```bash
# 分析最近3个提交并生成报告
python skills/change-model/scripts/generate-git-report.py \
  --range "HEAD~3..HEAD" \
  --title "新增筛选功能" \
  --type feature \
  --archive

# 输出到指定文件（不存档）
python skills/change-model/scripts/generate-git-report.py \
  --range "HEAD~1..HEAD" \
  --title "修复bug" \
  --type bugfix \
  --output /tmp/report.md
```

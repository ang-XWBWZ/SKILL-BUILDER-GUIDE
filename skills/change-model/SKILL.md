# Change Model — 变更模型技能模板

> **定位**：本技能是创建项目专属变更报告技能的**指南模板**，不是直接复制使用的成品。
>
> **使用方式**：参照本模板的四层架构（WHY/WHAT/HOW/VALIDATION），为你的项目生成专属的变更报告技能。

## 触发条件

- 创建变更报告技能 / 生成专属变更技能
- 生成变更报告 / DiffLog / Release Notes
- 从 Git 历史分析变更 / 从提交生成报告
- 归档变更记录 / 变更存档
- 查询历史变更 / 变更索引
- 询问"怎么写变更报告"/"怎么归档变更"
- 设计存档目录结构 / INDEX.md 结构

## 关联技能

- [技能构建指南](../skill-builder-guide/SKILL.md) — 技能创建方法论
- 调用链检查方法论见本文档[第四章](#四调用链检查方法论)，测试前验证数据流
- 调用链技能（代码级数据流追踪）模板见 `SKILL-BUILDER-GUIDE.md` 第 8.4 节

  - `generate-git-report.py` — Git 历史分析生成报告
  - `archive-report.py` — 手动报告归档

## 一、技能概述

### 1.1 什么是 Change Model

Change Model 是一种结构化的变更文档格式，将原始代码 diff 转化为：

- **WHY** — 变更背景与需求
- **WHAT** — 影响范围与风险
- **HOW** — 设计与实现
- **VALIDATION** — 验证与交付

### 1.2 解决的问题

| 问题 | Change Model 方案 |
|------|-------------------|
| Diff 碎片化 | 按因果链组织，而非按文件罗列 |
| 缺乏行为抽象 | 明确"条件 → 结果"的判断逻辑 |
| 新人理解困难 | 四层结构：背景 → 需求 → 设计 → 验证 |
| AI 无法推断意图 | 结构化、机器可读的章节设计 |
| 变更无法追溯 | 存档体系：按日期归档 + INDEX.md 索引 |
| 历史报告缺失 | Git 分析：从提交历史自动生成报告 |

### 1.3 本技能覆盖范围

**主体流程**：先写报告 → 变更实现 → 归档存档 → 更新索引

| 章节 | 内容 | 类型 |
|------|------|:----:|
| 一~七 | 变更报告方法论（四层架构、填写规范、调用链检查） | 方法论 |
| **八** | **变更存档设计**（目录结构、命名规范、索引结构） | 方法论 |
| **九** | **存档流程与工具**（手动/自动两种工作流 + 参考脚本） | 操作指引 |

**附属功能**：已有提交但未写报告时的补充能力

| 章节 | 内容 | 类型 |
|------|------|:----:|
| 附一 | 从 Git 历史生成变更报告 | 补充方法论 |
| 附二 | 与 CI/CD 集成 | 架构建议 |

**参考代码**：

| 文件 | 用途 |
|------|------|
| `scripts/generate-git-report.py` | 附属 — Git 历史分析生成报告 |
| `scripts/archive-report.py` | 主体 — 将已完成报告归入存档目录 |

### 1.4 与变更日志技能的区分

> 两个技能都涉及"变更"，但目的和粒度不同。

| 维度 | 变更模型 (change-model) | 变更日志 (diffs) |
|------|------------------------|-------------------|
| **视角** | 前瞻：变更发生前设计 | 回溯：变更发生后记录 |
| **粒度** | 每次需求/任务一个报告 | 每次提交/PR一条记录 |
| **内容** | WHY→WHAT→HOW→VALIDATION 完整因果链 | 改了什么、为什么改 |
| **时效** | 编码前完成设计，测试前完成验证 | 提交后自动/手动记录 |
| **用途** | 指导开发、风险控制 | 历史追踪、复盘审计 |
| **受众** | 开发者（执行变更的人） | 维护者、新人（理解历史的人） |

**简单记忆**：变更模型是"做之前的设计图"，变更日志是"做完后的记录本"。两者互补，可配合使用。

**填写要点**：

## 四、调用链检查方法论

### 4.1 什么是调用链检查

调用链检查是在测试前验证数据从入口到终点的完整路径，确保每个环节的正确性。

**核心目标**：

| 目标 | 说明 |
|------|------|
| 链路完整 | 从入口到终点无断裂 |
| 类型匹配 | 数据类型在每个环节正确传递 |
| 最终调用 | SQL/接口/消息等最终操作正确 |

### 4.2 检查流程

```
┌─────────────────────────────────────────────────────────┐
│  Step 1: 确定入口点                                      │
│  ├─ API 接口 / 页面事件 / 定时任务 / 消息消费            │
│  └─ 记录入参类型和校验规则                               │
├─────────────────────────────────────────────────────────┤
│  Step 2: 追踪中间环节                                    │
│  ├─ 接口层 → 业务层 → 数据层                              │
│  ├─ 记录每个环节的方法签名和返回类型                     │
│  └─ 检查类型是否匹配                                     │
├─────────────────────────────────────────────────────────┤
│  Step 3: 验证最终调用                                    │
│  ├─ SQL 语句 / 外部接口 / 消息队列 / 缓存写入            │
│  ├─ 检查参数是否正确传递                                 │
│  └─ 检查返回值是否正确处理                               │
├─────────────────────────────────────────────────────────┤
│  Step 4: 错误处理检查                                    │
│  ├─ 异常是否被捕获                                       │
│  ├─ 错误是否正确返回                                     │
│  └─ 降级逻辑是否存在                                     │
└─────────────────────────────────────────────────────────┘
```

### 4.3 检查项清单

| # | 检查项 | 检查内容 | 验证方法 |
|:-:|--------|----------|----------|
| 1 | **入口参数** | 参数类型、必填校验、格式校验 | 查看入口文件/接口定义 |
| 2 | **参数传递** | 参数在各层是否正确传递 | 追踪方法调用链 |
| 3 | **类型匹配** | 每个环节的入参/出参类型是否匹配 | 对比类型定义 |
| 4 | **空值处理** | 可选字段是否有默认值或空值检查 | 检查代码逻辑 |
| 5 | **最终调用** | SQL/接口/消息参数是否正确 | 检查最终实现 |
| 6 | **返回处理** | 返回值是否正确组装和返回 | 检查返回路径 |
| 7 | **异常处理** | 异常是否被捕获和处理 | 检查 try-catch |
| 8 | **降级逻辑** | 失败时是否有降级方案 | 检查 fallback |

### 4.4 常见问题与修复

| 问题类型 | 表现 | 原因 | 修复方案 |
|----------|------|------|----------|
| **链路断裂** | 方法未实现或未导出 | 忘记实现/访问修饰符错误 | 实现/导出方法 |
| **类型不匹配** | 运行时类型错误 | 类型转换缺失 | 添加类型转换 |
| **空值穿透** | NPE 异常 | 缺少空值检查 | 添加空值判断 |
| **参数丢失** | 参数未传递 | 方法签名变更后遗漏 | 同步更新调用方 |
| **SQL 错误** | 执行报错 | 字段名/类型不匹配 | 检查 SQL 和实体映射 |

## 八、变更存档设计

### 8.1 为什么需要存档

变更报告不应只是临时产物。存档让每次变更**可追溯、可复盘、可被 AI 后续理解**。

| 存档价值 | 说明 |
|----------|------|
| **可追溯** | 按日期索引，随时查看任何时间段的变更历史 |
| **可复盘** | 出问题时回溯"当时为什么这样改" |
| **AI 上下文** | 后续 AI 会话可读取历史报告，理解项目演进脉络 |
| **团队对齐** | 新人或协作者快速了解近期变动 |

### 8.2 存档时机

```
编码完成 → 调用链检查 → 测试验证 → 生成最终报告 → 📦 归档
```

| 阶段 | 操作 | 说明 |
|------|------|------|
| 变更前 | 创建报告草稿 | 填写 WHY / WHAT 部分 |
| 编码完成 | 填写 HOW 部分 | 实施完成后的文件清单和 diff |
| 测试前 | 填写 VALIDATION（调用链检查） | 确保链路正确 |
| 测试通过 | 完善测试结果 | 记录实际测试数据 |
| **归档** | **执行存档命令** | **最终版本落地到 docs/changes/** |

### 8.3 目录结构

```
{project}/
└── docs/
    └── changes/                  # 变更报告存档目录
        ├── INDEX.md              # 总索引（按时间 / 按类型双维度排列）
        ├── 2026-04-26/           # 按日期组织
        │   ├── 20260426-01-新增筛选功能.md
        │   └── 20260426-02-修复登录超时.md
        └── 2026-04-27/
            └── 20260427-01-重构支付模块.md
```

### 8.4 命名规范

```
{YYYYMMDD}-{NN}-{简短描述}.md

YYYYMMDD  — 归档日期（8位）
NN        — 当天序号（2位，从01开始，自动递增）
简短描述  — 功能名/变更主题（建议20字以内，英文用中划线连接）
```

**示例**：
- `20260426-01-新增筛选功能.md`
- `20260426-02-修复登录超时.md`
- `20260427-01-refactor-payment-flow.md`

### 8.5 INDEX.md 结构

索引文件提供两种排序维度：

```markdown
# 变更索引

## 按时间排序（最新在前）

| 日期 | 报告 | 类型 | 范围 | 状态 |
|------|------|:----:|------|:----:|

## 按类型排序

### 功能开发
- 2026-04-26 [新增筛选功能](2026-04-26/20260426-01-新增筛选功能.md)

### Bug 修复
- ...

### 重构

- ...
```

## 九、存档流程与工具

### 9.1 两种工作流

#### 手动存档（报告已写好）

```
编写报告 → 复核内容 → 执行存档命令 → 自动编号 → 更新索引
```

适用于：AI 会话中手动生成的结构化报告、已有变更文档需要归入统一索引。

#### 自动存档（从 Git 生成）

```
指定 Git 范围 → 脚本分析 → 自动生成报告 → 自动编号 → 更新索引
```

适用于：已提交代码但未写报告，需要快速补全变更记录。

### 9.2 参考实现脚本

> **所有脚本为参考实现**，需根据项目实际情况调整（模板内容、路径约定、评估规则等）。

| 脚本 | 用途 | 适用场景 |
|------|------|----------|
| `scripts/generate-git-report.py` | Git 历史分析 → 生成变更报告 | 补全历史记录、自动生成 |
| `scripts/archive-report.py` | 将已有报告文件归档 | 手动编写的报告归档 |

#### 生成报告脚本

```bash
# 分析最近3个提交并生成报告
python skills/{skill-name}/scripts/generate-git-report.py \
  --range "HEAD~3..HEAD" \
  --title "新增筛选功能" \
  --type feature \
  --archive

# 分析功能分支的全部变更
python skills/{skill-name}/scripts/generate-git-report.py \
  --range "main..feature-branch" \
  --title "重构支付模块" \
  --type refactor \
  --archive

# 输出到指定文件（不存档）
python skills/{skill-name}/scripts/generate-git-report.py \
  --range "HEAD~1..HEAD" \
  --title "修复bug" \
  --type bugfix \
  --output /tmp/report.md
```

#### 归档脚本

```bash
# 将已有的报告文件归档
python skills/{skill-name}/scripts/archive-report.py \
  path/to/report.md \
  --title "新增筛选功能" \
  --type feature \
  --scope api
```

### 9.3 存档校验清单

在报告进入最终存档前确认：

- [ ] 报告包含完整的四层结构（WHY / WHAT / HOW / VALIDATION）
- [ ] 调用链检查所有项目为 ✅
- [ ] 涉及的文件路径已与实际代码核对
- [ ] 回滚方案明确可执行
- [ ] 无敏感信息（密码、密钥、内部系统名 — 视项目而定）

### 9.4 索引维护守则

- **追加不覆盖** — 新条目追加到 INDEX.md，不覆盖已有记录
- **序号连续** — 同一天内的报告序号从 01 递增
- **类型准确** — 如实标记 feature / bugfix / refactor / config
- **状态更新** — 报告归档时默认 DONE，后续如需废弃可改为 DEPRECATED

## 附一：从 Git 历史生成变更报告（附属功能）

> **定位**：这是主流程的**补充能力**。当已有 Git 提交但未及时编写变更报告时，用于批量补全历史记录。正常情况下，应遵循"先写报告 → 再变更 → 归档"的主流程。

### 附1.1 核心步骤

```
① 确定分析范围 → ② 提取提交信息 → ③ 提取变更文件 → ④ 提取 diff → ⑤ 组合报告
```

| 步骤 | 目的 | 关键命令 |
|------|------|----------|
| **确定范围** | 界定要分析的提交区间 | `git log --oneline` 确认范围 |
| **提取提交** | 获取 hash/作者/日期/消息 | `git log <range> --format` |
| **提取文件** | 列出所有变更文件 | `git diff --name-status <range>` |
| **提取统计** | 新增/删除行数 | `git diff --shortstat <range>` |
| **提取 diff** | 完整代码变更 | `git diff <range>` |
| **组合报告** | 按四层模板生成 markdown | — |

### 附1.2 确定分析范围

| Git 范围 | 含义 | 适用场景 |
|----------|------|----------|
| `HEAD~1..HEAD` | 最近 1 个提交 | 单次小变更 |
| `HEAD~3..HEAD` | 最近 3 个提交 | 当天工作汇总 |
| `main..feature-x` | 分支的全部差异 | 功能分支合入前的报告 |
| `abc1234..def5678` | 两个指定提交之间 | 精确范围分析 |
| `v1.0.0..v1.1.0` | 两次发布之间 | Release Notes 生成 |

### 附1.3 自动评估规则

脚本基于以下通用规则自动评估影响等级（规则可按项目调整）：

```python
# 影响等级自动评估
if 文件数 < 3 and (新增行 + 删除行) < 50:
    level = "L0 — 无外部影响"
elif 文件数 <= 10 and (新增行 + 删除行) <= 200:
    level = "L1 — 内部变更"
else:
    level = "L2 — 外部契约变更（需人工确认是否为 L3）"
```

**L3 需人工判断**：涉及数据库迁移、配置文件变更、API 不兼容变更时，应手动升级为 L3。

### 附1.4 报告局限性声明

自动生成的报告存在以下局限，需人工补充：

| 局限 | 原因 | 补充建议 |
|------|------|----------|
| WHY 层内容简略 | 无法从 git 推断需求背景 | 人工补充变更背景和需求来源 |
| 调用链检查空白 | 无法自动执行代码级验证 | 测试前手动完成调用链检查 |
| 判断逻辑缺失 | git 消息通常只含"做了什么" | 人工补充条件 → 结果判断逻辑 |
| 风险项基于数量 | 不理解业务语义 | 人工审核并补充业务风险 |
| 关联信息缺失 | 无 PR/需求单链接 | 人工补充相关链接和文档引用 |

**最佳实践**：脚本生成骨架 → 人工补充 WHY 层和调用链检查 → 复核风险 → 最终存档。

## 附二：与 CI/CD 集成（延伸参考）

### 附2.1 概念流程

> 以下为架构级建议，供项目集成时参考。具体实现取决于使用的 CI/CD 平台。

```
PR 合并到 main
    │
    ▼
CI 触发
    │
    ├─ 1. git log 提取合并提交的范围
    ├─ 2. 运行 generate-git-report.py --archive
    ├─ 3. 将 docs/changes/ 变更提交到仓库
    └─ 4. 发送通知（可选）
```

### 附2.2 触发时机建议

| 触发事件 | 自动化程度 | 说明 |
|----------|:----------:|------|
| PR 合并 | 全自动 | 合并后自动生成报告并归档 |
| 版本发布 | 全自动 | 生成 Release Notes 格式的报告 |
| 手动触发 | 半自动 | 提供命令供开发者按需执行 |
| 定期汇总 | 全自动 | 每周/每月生成周期汇总报告 |

### 附2.3 集成要点

- 脚本为纯标准库实现，无需安装依赖，CI 中可直接执行
- 需确保 CI 环境有完整的 git 历史（`fetch-depth: 0`）
- 存档后的 `docs/changes/` 建议提交回仓库，保持单一事实源
- 敏感检查：CI 中应增加一道检查，确保报告不包含密钥/令牌等

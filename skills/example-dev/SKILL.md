# 开发规范技能模板

> **定位**：本技能是创建项目专属开发规范技能的**通用指南模板**，不是直接复制使用的成品。
>
> **使用方式**：AI 扫描项目代码 → 提取各层标准写法 → 参照本模板结构 → 生成项目专属的开发规范技能。

## 触发条件

- 创建开发规范技能
- 询问如何编写技术栈、代码规范
- 询问编码风格、命名规范
- 需要代码分层规范指导
- 需要标准写法识别方法

## 二、代码分层规范

### 2.1 分层架构（通用模型）

```
┌─────────────────────────────────────────────────────────┐
│  接口层 (API/入口层)                                     │
│  ├─ 请求参数校验与格式转换                                │
│  ├─ 统一响应封装                                         │
│  └─ 错误处理与异常转换                                   │
├─────────────────────────────────────────────────────────┤
│  业务层 (业务逻辑层)                                     │
│  ├─ 核心业务逻辑                                         │
│  ├─ 事务/一致性管理                                       │
│  ├─ 数据组装与转换                                       │
│  └─ 外部服务编排                                         │
├─────────────────────────────────────────────────────────┤
│  数据层 (数据访问层)                                     │
│  ├─ 持久化操作                                           │
│  ├─ 查询构建                                             │
│  └─ 缓存操作                                             │
├─────────────────────────────────────────────────────────┤
│  集成层 (可选，按项目实际存在)                             │
│  ├─ 远程服务调用                                         │
│  ├─ 消息队列处理                                         │
│  ├─ 定时任务                                             │
│  └─ 第三方 SDK 封装                                      │
└─────────────────────────────────────────────────────────┘
```

### 2.2 各层职责边界

| 层级 | 职责 | 禁止事项 |
|------|------|----------|
| **接口层** | 参数校验、格式转换、统一响应 | 业务逻辑、直接访问数据库 |
| **业务层** | 业务逻辑、一致性管理、数据组装 | 直接处理请求/响应对象、拼接查询语句 |
| **数据层** | 持久化操作、查询构建 | 业务逻辑判断 |
| **集成层** | 跨服务调用、异步消息处理 | 核心业务逻辑实现 |

| 业务层目录 | 放置业务逻辑的目录 | `{目录名}` |

| 数据层命名 | 数据访问文件的命名模式 | `{命名模式}` |
| 集成层目录 | 放置外部集成的目录 | `{目录名}` |

| 检查项 | 扫描内容 | 记录格式 |
|--------|----------|----------|
| 路由声明 | 路由配置方式（注解/配置文件/函数） | `{方式}` |

| 响应封装 | 统一响应结构（类/函数/中间件） | `{类名或函数名}` |
| 错误处理 | 错误处理方式（全局处理/逐方法处理） | `{方式}` |

**伪代码模板（通用）**：

```

    ├─ 参数校验
    ├─ 调用业务层
    └─ 返回统一响应格式
```

### 3.2 业务层标准写法

| 检查项 | 扫描内容 | 记录格式 |

| 外部调用 | 调用其他服务的方式 | `{SDK/远程调用/消息}` |

**伪代码模板（通用）**：

```

    ├─ 一致性边界声明
```

## 五、兼容写法识别

### 5.1 什么需要记录

> AI 扫描代码时，识别与主流写法不同但因历史原因需要保留的写法。

| 场景 | 示例 | 记录说明 |
|------|------|----------|
| 遗留格式 | 老接口返回格式与新接口不同 | 保留，标注适用范围 |
| 特殊优化 | 特定场景的高性能处理方式 | 保留，记录原因 |
| 业务特例 | 某个模块的独特事务处理 | 保留，标注适用条件 |

### 5.2 记录模板

```markdown
## 兼容写法记录

| 模块 | 兼容写法 | 原因 | 标准替代 |
|------|----------|------|----------|
| {模块A} | {特殊写法} | {历史原因/性能要求} | {如有标准替代则列出} |
```

## 六、编码风格形成流程

```
① 项目分析
    ├─ 识别项目类型（Web服务/数据处理/API网关/其他）
    ├─ 统计技术栈（语言、框架、数据库）
    └─ 梳理模块结构
       │
       ▼
② 代码扫描（派 Haiku — L0 任务）
    ├─ 扫描各层目录结构
    ├─ 收集各层代码样本
    └─ 提取配置文件
       │
       ▼
③ 风格提取
    ├─ 识别各层标准写法
    ├─ 提取命名模式
    ├─ 识别兼容写法及原因
    └─ 记录特殊约定
       │
       ▼
④ 文档生成
    ├─ 生成技术栈表
    ├─ 生成分层架构说明
    ├─ 填充各层标准写法模板
    └─ 生成命名规范表
       │
       ▼
⑤ 用户确认
    ├─ 技术栈版本是否正确
    ├─ 标准写法是否符合预期
    ├─ 兼容写法原因是否准确
    └─ 是否有补充文档合并
```

## 七、生成专属技能

**流程**：

```
用户请求 → AI 分析项目 → 扫描代码 → 提取风格 → 生成专属 SKILL.md
```

**AI 执行步骤**：

1. 分析项目类型和技术栈
2. 派 Haiku 扫描各层代码样本（L0 任务）
3. 提取标准写法和兼容写法
4. 按本模板结构填充项目实际内容
5. 生成项目专属的开发规范 SKILL.md
6. 用户确认或提供文档调整

### 7.1 用户文档合并

> 如果用户提供编码规范文档，优先以用户文档为准。

```
1. 读取用户提供的编码规范文档
2. 与扫描结果对比差异
3. 以用户文档为准，补充扫描发现的实际情况
4. 记录差异项，供用户确认
```
